# 執筆ルール

## 1. 文体

### トーン

<!-- /setup で語りかけの人格を設定する（例: 先輩エンジニア、メンター、講師） -->

- **本文**: 落ち着いた敬体で丁寧に解説する。過度に個人的な語りかけ（「率直に言うと」等）は避ける
- **🧠 [人格名]はこう考える**: 実体験ベースの語りをここに集中させる

### 用語・表記

- 敬体（です・ます調）で統一。読者への呼びかけは「あなた」
- ダッシュ記号（`——`、`—`、`–`）は使わない
- {{TOPIC}} の用語・コード例・コマンド例は公式ドキュメントの表記に従う。範囲外の内容は WebSearch / WebFetch で裏付ける
- 時間経過で変わる情報は最新情報を取得し「○○時点」と日付を明記
- 外部ソースを参照する場合は出典を明記する
- Markdown の `**text**` は閉じ `**` 直後にスペースなしで日本語が続くと適用されない。`**○○** の場合` のようにスペースを入れる

<!-- /setup で用語テーブルを記入 -->

| 表記 | 備考 |
|---|---|
| セクション | 教材本文ではカタカナ表記。内部ドキュメントでは英語 |

### 絵文字

| 絵文字 | 用途 |
|---|---|
| 🎯 | 学習目標（Section 冒頭） |
| ✨ | まとめ（Section 末尾） |
| 💡 | TIP・補足 |
| ⚠️ | 注意事項・よくあるエラー |
| 📝 | ノート・定義・補足説明 |
| 🔑 | 重要概念・キーポイント |
| 🏃 | ハンズオン手順・実践 |
| 🧠 | 経験者の視点・思考プロセス |
| 🔍 | 見極めチェック |
| 📌 | 操作前の確認事項 |
| 📖 | この Chapter の進め方（2層以上の場合） |

## 2. 構造

### Section の種類

| 種類 | 内容 |
|---|---|
| **概念** | 意義・仕組み・使い方を解説。コード生成なし |
| **ハンズオン** | 概念で学んだ機能を実践。冒頭に概念 Section への逆リンク |
| **混合** | 概念を学びながらすぐに手を動かす |

### 見出しの設計

**Why → What → How**（抽象→具体）の流れで組む。

### テンプレート

すべての Section に共通する骨格:

```markdown
# [セクション番号] セクションタイトル

## 🎯 このセクションで学ぶこと

- 学習目標1
- 学習目標2

[1文で Section の流れを示す]

---

## 導入: [課題・痛みから入る見出し]

### 🧠 [人格名]はこう考える

> [実務の経験に基づいて自由に語る]

---

## [本文の見出し]

## ✨ まとめ

- ポイント1
- ポイント2

---

次のセクションでは、[次の内容への接続]。
```

**種類ごとの追加要素**:

| 種類 | 追加要素 |
|---|---|
| **概念** | 2層以上の場合、親グループの最初の Section のみ、タイトル直後にグループ全体像（概要・セクション一覧・📖 進め方）を含める |
| **ハンズオン** | 冒頭に `📝 このハンズオンで使う機能: [機能名]（[セクション名] で学習）`。本文を 🏃 Step 形式で構成。`✅ 完成チェックリスト` と `🔍 見極めチェック` を含める。長いコードは `<details>` で折りたたむ |
| **混合** | 本文の後に `🏃 実践` パート。コード生成を伴う場合は `🔍 見極めチェック` を含める |

**🔍 見極めチェック**（ハンズオン・混合でコード生成を伴う場合）:

```markdown
## 🔍 見極めチェック

> 🧠 [人格名]の思考: 「[なぜこの観点で検証するのか]」

- [ ] **正しさ**: [具体項目]
- [ ] **品質**: [具体項目]
- [ ] **安全性**: [具体項目]

> 🔑 この Section では特に「○○」に注目してください。
```

3観点すべてにチェック項目を列挙。該当なしの観点は明記する。

**エラーフォーマット**:

> ⚠️ **よくあるエラー**: [概要]
>
> ```
> [エラーメッセージ例]
> ```
>
> **原因**: [なぜ発生するか]
>
> **対処法**: [解決手順]

## 3. コンテンツ

### コードの見せ方

| 種類 | 言語指定 |
|---|---|
| ターミナルコマンド | `bash` |
| ツールへの入力（プロンプト） | なし（行頭に `>` を付ける） |
| 生成コード・ファイル内容 | 適切な言語（`php`, `json` 等） |
| コマンド実行結果 | なし |

- 生成コードには「あなたの環境では異なるコードが生成されます」と前置き（初回は丁寧に、以降は簡潔に）
- コードブロック上部にファイルパスをコメントで明示
- diff 表示は使わない（テキストで説明し変更後のコードを示す）
- プロンプト例・コード例は要件レベルで書き、特定のファイル名・コード構造に依存しない

### 解説の深さ

- 前提知識は基本説明を省略。ただし {{TOPIC}} と組み合わせたときの注意点は丁寧に
- {{TOPIC}} 固有の概念は初出時に丁寧に解説し、2回目以降は初出 Section への参照
- 同じ概念を別の箇所で繰り返さない
- 導入は「これがないと何が困るか」（課題）から入る
- 処理の流れやサイクルは ASCII フロー図で視覚化する
- つまずきポイントを先回りする（⚠️ で説明）
- 先の Section で扱う内容は「今は概要だけ把握すれば十分です」と明記する
- 初回リリースでは画像を含めず `<!-- TODO: 画像追加 - ○○の画面 -->` でプレースホルダーを残す
- 1 Section あたり**最低 4,000 文字**。説明を省略せず丁寧に書く
