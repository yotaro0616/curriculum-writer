# 最新情報の差分チェック

{{TOPIC}} の最新ドキュメント・Changelog と教材の内容を照合し、更新が必要な箇所を報告する。自動修正はしない。

**入力**: $ARGUMENTS

**解釈ルール**:
- `Part番号-Chapter番号` 指定時: その Chapter の参考文献URL・コマンドのみチェック
- 未指定時: 全 Chapter を対象にチェック

**前提**: `OUTLINE.md` の参考文献URLとスコープ定義を照合基準とする。Chapter ファイルが存在する場合のみ既存コンテンツの鮮度チェックを行う。

## ステップ1: 情報源の取得

CLAUDE.md の MAP セクションに記載された公式ドキュメント URL を読み取り、以下を WebFetch で**並列に**取得する:

1. 公式ドキュメントのページ一覧（llms.txt 等が提供されている場合）
2. Changelog（直近3バージョン重点）
3. `OUTLINE.md` の参考文献URL一覧を読み取る

WebFetch 失敗時は失敗URLと理由を記載し、取得できた情報源のみで続行。全取得不可なら報告して終了。

## ステップ2: ドキュメント構成の差分チェック

取得したドキュメント一覧と `OUTLINE.md` を照合:

1. **新規ページ**: ドキュメント一覧にあるが教材にない → 既存 Chapter への追加 or 新規 Chapter 候補
2. **削除・統合**: 参考文献にあるがドキュメント一覧にない
3. **スコープ判定**: 新規ページが既存 Section に該当するか、新 Chapter/Section が必要か

## ステップ3: Changelog との差分チェック

Changelog の変更と教材を照合:

1. API・コマンド・設定の変更（廃止、新追加、構文変更）
2. 新機能の追加（教材スコープ内）
3. UI・操作手順の変更
4. 破壊的変更（最優先で報告）

Chapter ファイルが存在する場合は内容と直接照合。未作成の場合は `OUTLINE.md` の設計と照合。

## ステップ4: 参考文献URLの有効性チェック

ドキュメント一覧の URL に含まれるか照合し、含まれない URL のみ WebFetch で確認する。リダイレクトがあれば記録。

## ステップ5: 結果の報告

以下を報告: チェック実施日、Changelog 最新バージョン、情報源の取得状況、破壊的変更、新機能・変更（未反映）、ドキュメント構成の変更、既存コンテンツの更新箇所、参考文献URLの状態、推奨アクション。

各項目に優先度（高: 既存解説が誤りになる / 中: スコープ内の主要新機能 / 低: マイナー改善）を付与する。

---

## メンテナンス運用方針

### 定期チェック

- `/check-updates` を**月1回**実行し、公式ドキュメント・Changelog との差分を確認する
- {{TOPIC}} のメジャーアップデート時は即座に実行する

### 更新フロー

1. `/check-updates` で差分を検出
2. 破壊的変更（優先度「高」） → 即座に `/write` で修正
3. 新機能・中程度の変更 → 次回の更新サイクルで対応
4. 修正後、`/review` で整合性を確認

Part・Chapter 構成の変更（追加・削除・並べ替え）は教材全体に影響するため、`/setup` を使用すること。
