# 教材のレビュー

問題点は自動修正せず、すべて報告のみとする。

**入力**: $ARGUMENTS

## スコープの解釈

| 入力形式 | 対象 | 例 |
|---|---|---|
| `Part番号` or `番号` | Part 全体 | `1`, `Part 2` |
| `Part番号-Chapter番号` | Chapter | `2-1`, `Chapter 2-3` |
| `Part番号-Chapter番号-Section番号` | Section | `2-3-5`, `Section 1-1-2` |

## 0. 並列化（Part 全体または複数 Part が対象の場合）

対象が Part 全体以上のスコープの場合、Part ごとにサブエージェントを並列で起動する。各サブエージェントには以下を渡す:

- 対象 Part の OUTLINE.md の該当セクション
- 対象 Part の全 Section ファイルパス
- `.claude/rules/writing.md` の全ルール（テンプレート・文体・コンテンツ規則）を漏れなく伝える
- レビュー観点（セクション2「単体レビュー」の全チェック項目）
- 出力フォーマット（セクション4「出力」のテーブル形式）

各サブエージェントの結果を統合し、横断整合性チェック（セクション3）はメインで実行する。

## 1. 準備

1. `OUTLINE.md` を Read し、対象スコープのゴール・Section 構成を確認する
2. 対象スコープの全 Section ファイルを Read する
3. 前後のスコープの Section ファイルを Read する（未執筆の場合は OUTLINE.md で代替）

## 2. 単体レビュー

以下の6観点で検証する。

### テンプレート準拠

`.claude/rules/writing.md` のテンプレートに準拠しているか:
- 🎯 このセクションで学ぶこと / ✨ まとめ が揃っているか
- `---` 区切りが適切に使われているか
- ハンズオン Section に 🔍 見極めチェック（3観点 MECE）があるか
- ハンズオン Section 冒頭に使用機能への逆リンクがあるか
- 混合 Section に 🏃 実践パートがあるか
- 絵文字が定義済みの種類の範囲内か

### 設計との整合

`OUTLINE.md` の設計と実際の内容を照合:
- Section 構成・ゴールが OUTLINE.md と一致しているか
- Section の種類（概念/ハンズオン/混合）が守られているか

### 正確性

- {{TOPIC}} の用語が公式ドキュメントの表記に従っているか
- コードブロックの言語指定が正しいか

> 📝 公式ドキュメントとの詳細な照合（コマンド・機能説明の最新性）は `/check-updates` スキルの責務。このスキルでは構造・スタイル・出典の検証に集中する。

### スタイル準拠

`.claude/rules/writing.md` に準拠しているか:
- 敬体（です・ます調）で統一されているか
- {{TOPIC}} の用語が公式ドキュメントの表記に従っているか
- 読者への呼びかけが「あなた」か

### 実践フォロー可能性

🏃 実践・ハンズオンを含む Section（混合・ハンズオン型）が対象。読者が記載内容だけで手を動かして最後まで進められるかを検証する:

- **前提条件の明記**: 実践の冒頭に 📌 ノートで前提条件（環境起動、ツールのインストール、ログイン情報など）が明記されているか。セッションをまたぐ場合に環境停止の可能性を考慮しているか
- **手順の整合性**: コマンドのフラグと説明が矛盾していないか。Step の順序に飛躍がないか。操作結果の説明が実際の挙動と一致するか
- **Chapter 間の依存**: 前の Chapter の成果物（ブランチ、生成ファイル、設定変更など）への依存が明示されているか。ブランチの切り替え・統合手順が説明されているか
- **外部リソースのフォールバック**: マーケットプレイス、API キー、GitHub 権限など外部依存がある場合、利用できないときの代替パスが示されているか
- **OS 依存の代替**: macOS 固有のコマンドに Linux / WSL の代替手順があるか
- **資格情報の提供**: テストアカウントでのログインが必要な場合、メールアドレス・パスワードがカリキュラム内のどこかで明示されているか
- **実践プロジェクトとの依存関係**: 前の Chapter で変更したコード（ブランチ、ファイル変更）が後の Chapter の前提になっている場合、その依存が明示されているか
- **コード参照の実在性**: 🏃 実践で言及するファイルパス・メソッド名が実プロジェクトに存在するか

### リンク・参照の検証

- **Section 間リンク**: 相対リンク（`../../part-02_...` 等）の参照先ファイルが実在するか
- **Section 番号の整合**: 本文中の「X-X-X で学んだ」という参照が、実際の Section 番号・タイトルと一致しているか
- **OUTLINE.md との番号一致**: OUTLINE.md で定義された Section 番号と、実際のファイル名・本文中の番号がずれていないか

## 3. 横断整合性チェック

> 対象スコープ外に既存コンテンツが存在する場合のみ実行する。

- **用語の一貫性**: 同じ概念が異なる言葉で説明されていないか
- **文体の統一**: 説明密度・語りかけ方にばらつきがないか
- **接続の自然さ**: Section 間・Chapter 間の前後関係が自然か

## 4. 出力

```
**[スコープ] レビュー結果**: 指摘 X 件（🔴 X / 🟡 X / 🔵 X）

## 修正アクションリスト

| # | 重要度 | ファイル | 箇所 | 修正内容 |
|---|---|---|---|---|
```

重要度: 🔴 技術的誤り / 🟡 設計不整合 / 🔵 軽微なスタイル問題
