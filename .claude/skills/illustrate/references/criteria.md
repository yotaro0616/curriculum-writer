# 画像挿入の判断基準

教材の Section をスキャンして「AI 画像が本当に必要な箇所」を特定するための基準。

## 5つの判断観点

以下のいずれかに該当する場合、AI 画像の挿入を検討する。

| # | 観点 | 説明 | 具体例 |
|---|------|------|--------|
| 1 | **メンタルモデルの可視化** | 抽象的な概念を直感的なイメージで伝えたい | イベントループ、仮想 DOM、依存性逆転 |
| 2 | **鳥瞰アーキテクチャ** | 複雑なシステムの全体像を「地図」として俯瞰したい | AWS 構成、通知アーキテクチャ |
| 3 | **Before/After の対比** | 技術導入の「なぜ」を課題→解決で実感させたい | Fat Controller → Clean Architecture |
| 4 | **比喩・アナロジー** | 🧠 セクションで使われている比喩を絵にしたい | Props = 一方通行の荷物配達 |
| 5 | **パラダイムシフト** | 読者の思考の転換点となる概念を1枚で示したい | ページ思考 → コンポーネント思考 |

## Mermaid との判断フローチャート

ある概念を図にすべきか迷ったときの判断基準:

1. **正確な処理順序やデータの流れを示したい？** → **Mermaid**（既にある可能性が高い）
2. **「こういう感覚で捉えてほしい」という直感を伝えたい？** → **AI 画像**
3. **Before/After の対比で「なぜ必要か」を実感させたい？** → **AI 画像**
4. **初見の読者が Section を開いたとき、これから何を学ぶか一瞬で掴んでほしい？** → **AI 画像**
5. **頻繁に更新される可能性がある？** → **Mermaid**（テキストで管理できる）

**一言で整理:**
- Mermaid = **How it works**（仕組みの正確な図解）→ 本文中に配置
- AI 画像 = **How to think about it**（概念の直感的な掴み）→ 導入 🧠 直後に配置

## 量の目安

全 Section に画像を入れる必要はない。「これがないと読者が本当につまずく概念」に絞る。

- 目安: 1 Chapter あたり 1〜2 枚
- 読者にとって完全に新しい概念（前提知識にない領域）を優先する
- 既存の Mermaid 図で十分に伝わる箇所はスキップする

## scan 結果の報告フォーマット

```markdown
### [Section 番号] [Section タイトル]

- **挿入位置**: 導入 🧠 直後
- **観点**: [5つの観点のどれか]
- **理由**: [なぜ画像が必要か、1〜2文]
- **画像名**: `image-name`
- **プロンプト案**: [画像に描く内容の概要]
```
