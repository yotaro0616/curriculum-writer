## 導入: `require` も `use` もない？ JavaScript の依存管理

PHP では、別ファイルのクラスを使うときに `use App\Models\User;` と書きます。パッケージを追加するときは `composer require` を実行し、`vendor/` にライブラリがインストールされます。配列を変換するときは `array_map()` を呼び出します。

JavaScript にも同じ目的を果たす仕組みがありますが、構文も管理方法もかなり異なります。LMS のフロントエンドコードを開くと、ファイルの先頭にこのような行が並んでいます。

```tsx
// frontend/src/app/v2/user/[workspaceId]/(main)/curriculums/[curriculumId]/chapters/[chapterId]/sections/[sectionId]/Client.tsx
import MarkDownToHTML from '@/components/v2/elements/MarkDownToHTML'
import { PageLayout } from '@/components/v2/templates/PageLayout'
import { ACCOUNT_TYPE } from '@/constants/v2/accountType'
import type { FetchPublicCurriculumHttpDocument } from '@/features/v2/curriculum/api/fetchPublicCurriculum'
import { useRouter } from 'next/navigation'
import { useCallback, useEffect, useRef } from 'react'
```

`use` ではなく `import`。`namespace` ではなく `export`。PHP とは異なるキーワードですが、考え方の根本は同じです。このセクションでは、これらの構文を PHP との対比で整理し、LMS のコードを読む力を身につけます。

### 🧠 先輩エンジニアはこう考える

> LMS のフロントエンドには 70 以上の npm パッケージが入っていて、ファイル数も数百を超えます。モジュールシステムを理解していないと、「この関数はどこから来ているのか」「このライブラリは何のために入っているのか」がまったく追えません。逆に、`import` / `export` の仕組みと `package.json` の読み方さえ分かれば、初めて見るファイルでもすぐに依存関係を辿れます。配列操作の `map` / `filter` も、React でリストを描画するときに毎回使うので、ここで確実に押さえておくと後の Chapter がスムーズです。

## ES Modules（import / export）

### なぜモジュールシステムが必要か

PHP では `namespace` によってクラスや関数を論理的にグループ化し、`use` で他のファイルから参照します。JavaScript にも同じニーズがあります。ファイルごとに変数や関数を分離し、必要なものだけを明示的に公開・取得する仕組みが **ES Modules**（ESM）です。

🔑 ES Modules は ECMAScript 2015（ES6）で標準化されたモジュールシステムです。それ以前は `require()` を使う CommonJS という仕組みが Node.js で使われていましたが、現在のフロントエンド開発では ES Modules が標準です。LMS でも ES Modules を使用しています。

### PHP との対比で理解する基本構文

PHP と JavaScript のモジュール機構を並べて比較しましょう。

**PHP の場合:**

```php
// app/Models/User.php
namespace App\Models;

class User extends Model
{
    // ...
}
```

```php
// app/Http/Controllers/UserController.php
use App\Models\User;

class UserController extends Controller
{
    public function index()
    {
        $users = User::all();
    }
}
```

**JavaScript の場合:**

```javascript
// models/user.js
export class User {
    constructor(name) {
        this.name = name;
    }
}
```

```javascript
// controllers/userController.js
import { User } from './models/user.js';

const user = new User('太郎');
```

対応関係を整理すると、以下のようになります。

| PHP | JavaScript (ES Modules) | 役割 |
|---|---|---|
| `namespace` | ファイル自体がスコープ | 名前空間の定義 |
| `use App\Models\User` | `import { User } from '...'` | 他ファイルの参照 |
| `class` / 関数をファイルに書く | `export` を付ける | 外部への公開 |

PHP ではファイル内のクラスや関数はデフォルトで外部からアクセスできますが、JavaScript の ES Modules では **`export` を付けないものは外部からアクセスできません**。これは大きな違いです。ファイル内の変数や関数はデフォルトでプライベートであり、公開したいものだけを明示的に `export` します。

### 名前付きエクスポートとデフォルトエクスポート

JavaScript の `export` には2種類あります。

**名前付きエクスポート（Named Export）:**

```javascript
// utils.js
export const TAX_RATE = 0.1;
export function calculateTax(price) {
    return price * TAX_RATE;
}
```

```javascript
// main.js
import { TAX_RATE, calculateTax } from './utils.js';
```

名前付きエクスポートでは、`export` したときの名前をそのまま `{}` で囲んでインポートします。PHP の `use App\Utils\calculateTax;` に近い感覚です。

**デフォルトエクスポート（Default Export）:**

```javascript
// UserCard.js
export default function UserCard({ name }) {
    return `<div>${name}</div>`;
}
```

```javascript
// main.js
import UserCard from './UserCard.js';
// 名前を変えてインポートすることもできる
import MyCard from './UserCard.js';
```

デフォルトエクスポートは1ファイルにつき1つだけ定義でき、インポート時に `{}` が不要です。任意の名前で受け取れるのが特徴です。

💡 LMS では、React コンポーネント（1ファイル1コンポーネント）にはデフォルトエクスポート、定数や型定義には名前付きエクスポートを使うパターンが多く見られます。

### 再エクスポート（Re-export）

複数のファイルに分散した `export` を1つのファイルにまとめ直すパターンです。PHP にはない JavaScript 特有の仕組みです。

LMS の実際のコードを見てみましょう。

```typescript
// frontend/src/constants/v1/index.ts
export * from './env'
```

```typescript
// frontend/src/components/v1/kit/index.ts
export * from './elements'
export * from './forms'
```

`export * from './env'` は、`./env` ファイルが名前付きエクスポートしているすべてのものを、このファイルからも再エクスポートするという意味です。利用する側は個別のファイルパスを知らなくても、`index.ts` からまとめてインポートできます。

```typescript
// こう書く代わりに
import { API_URL } from '@/constants/v1/env';
import { APP_URL } from '@/constants/v1/env';

// こう書ける
import { API_URL, APP_URL } from '@/constants/v1';
```

PHP で言えば、1つの Facade や Service Provider を通して複数のクラスにアクセスできるようにするパターンに近い発想です。

### `import type` パターン

LMS のコードでは、通常の `import` に加えて `import type` という構文が頻繁に登場します。

```typescript
// frontend/src/type/v2/index.ts
import type { ACCOUNT_TYPE } from '@/constants/v2/accountType'
import type { FetchCurrentEmployeeAsEmployeeHttpDocument } from '@/features/v2/employee/api/fetchEmployeeMe'
```

```typescript
// frontend/src/store/v2/actor-store.ts
import type { Actor, ActorType } from '@/type/v2'
```

`import type` は TypeScript の機能で、「この import は型情報だけを取得しており、実行時のコードには含めない」という意味です。ビルド時に完全に削除されるため、バンドルサイズ（ブラウザに送られるファイルの大きさ）に影響しません。

今は「型だけをインポートする特別な構文がある」ということだけ把握しておけば十分です。TypeScript の型システムについては次の Chapter で詳しく学びます。

## npm / Yarn パッケージ管理

### Composer との対比

PHP で外部ライブラリを使うときは Composer を使います。JavaScript には **npm**（Node Package Manager）があり、LMS では npm の互換ツールである **Yarn** を使用しています。

| 概念 | PHP (Composer) | JavaScript (npm / Yarn) |
|---|---|---|
| パッケージ定義ファイル | `composer.json` | `package.json` |
| ロックファイル | `composer.lock` | `yarn.lock` / `package-lock.json` |
| インストール先 | `vendor/` | `node_modules/` |
| インストールコマンド | `composer install` | `yarn install` / `npm install` |
| パッケージ追加 | `composer require パッケージ名` | `yarn add パッケージ名` / `npm install パッケージ名` |
| 開発用パッケージ追加 | `composer require --dev パッケージ名` | `yarn add -D パッケージ名` |
| パッケージ公開レジストリ | Packagist | npm レジストリ |

考え方はほぼ同じです。`composer.json` に相当するのが `package.json`、`vendor/` に相当するのが `node_modules/` です。

### package.json の構造

LMS の `package.json` の主要部分を見てみましょう。

```jsonc
// frontend/package.json
{
  "name": "coachtech-lms-next",
  "version": "0.1.0",
  "private": true,
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "lint": "run-p -l -c --aggregate-output lint:*",
    "fix": "run-s fix:prettier fix:eslint"
  },
  "dependencies": {
    "@heroui/react": "^2.7.10",
    "next": "^14.2.3",
    "react": "^18.3.1",
    "react-hook-form": "^7.51.3",
    "swr": "^2.2.4",
    "zustand": "^5.0.1"
  },
  "devDependencies": {
    "@typescript-eslint/eslint-plugin": "^6.20.0",
    "eslint": "^8.56.0",
    "prettier": "^3.2.5"
  }
}
```

`composer.json` の `require` と `require-dev` に対応するのが `dependencies` と `devDependencies` です。

| フィールド | Composer での対応 | 用途 |
|---|---|---|
| `dependencies` | `require` | 本番環境でも必要なパッケージ |
| `devDependencies` | `require-dev` | 開発時のみ必要なパッケージ（リンター、テストツール等） |
| `scripts` | `composer scripts` | よく使うコマンドのエイリアス |

LMS の `dependencies` には 70 以上のパッケージが登録されています。React、Next.js、UI ライブラリ（HeroUI、MUI）、状態管理（Zustand）、データ取得（SWR）、フォーム管理（React Hook Form）など、フロントエンド開発に必要なライブラリ群が並んでいます。

### バージョン指定の読み方

`"next": "^14.2.3"` の `^` はバージョン範囲を示す記号です。Composer でも同じ記号を使うため、馴染みがあるでしょう。

| 記号 | 意味 | 例 |
|---|---|---|
| `^14.2.3` | メジャーバージョンを固定（14.x.x の範囲） | Composer と同じ |
| `~14.2.3` | マイナーバージョンを固定（14.2.x の範囲） | Composer と同じ |
| `14.2.3` | 完全一致 | Composer と同じ |

⚠️ **注意**: `node_modules/` は `.gitignore` に含まれており、Git には追跡されません。`vendor/` を Git に含めないのと同じ理由です。`yarn.lock`（または `package-lock.json`）はチーム全員が同じバージョンのパッケージを使うために Git に含めます。これも `composer.lock` と同じ考え方です。

## 配列の高階関数

### PHP の配列関数との根本的な違い

PHP では配列を変換するときに `array_map()` や `array_filter()` といったグローバル関数を使います。JavaScript では、配列オブジェクトの **メソッド** として呼び出します。

```php
// PHP
$numbers = [1, 2, 3, 4, 5];
$doubled = array_map(fn($n) => $n * 2, $numbers);
$evens = array_filter($numbers, fn($n) => $n % 2 === 0);
```

```javascript
// JavaScript
const numbers = [1, 2, 3, 4, 5];
const doubled = numbers.map((n) => n * 2);
const evens = numbers.filter((n) => n % 2 === 0);
```

PHP では `array_map(コールバック, 配列)` と「関数が主役」の書き方ですが、JavaScript では `配列.map(コールバック)` と「配列が主役」の書き方になります。この違いが後述するメソッドチェーンの読みやすさに大きく影響します。

🔑 **高階関数** とは、関数を引数として受け取る関数のことです。`map` や `filter` は引数にコールバック関数を受け取るため、高階関数と呼ばれます。セクション 2-1-3 で学んだアロー関数がここで活きてきます。

### 主要な配列メソッド

JavaScript でよく使われる配列メソッドを PHP の対応関数と並べて整理します。

**`map` : 各要素を変換して新しい配列を返す**

```php
// PHP
$users = [['id' => 1, 'name' => '太郎'], ['id' => 2, 'name' => '花子']];
$names = array_map(fn($user) => $user['name'], $users);
// ['太郎', '花子']
```

```javascript
// JavaScript
const users = [{ id: 1, name: '太郎' }, { id: 2, name: '花子' }];
const names = users.map((user) => user.name);
// ['太郎', '花子']
```

**`filter` : 条件に合う要素だけを残した新しい配列を返す**

```php
// PHP
$numbers = [1, 2, 3, 4, 5];
$evens = array_filter($numbers, fn($n) => $n % 2 === 0);
// [2, 4]（キーが維持される点に注意）
```

```javascript
// JavaScript
const numbers = [1, 2, 3, 4, 5];
const evens = numbers.filter((n) => n % 2 === 0);
// [2, 4]
```

**`reduce` : 配列を1つの値に集約する**

```php
// PHP
$numbers = [1, 2, 3, 4, 5];
$sum = array_reduce($numbers, fn($carry, $n) => $carry + $n, 0);
// 15
```

```javascript
// JavaScript
const numbers = [1, 2, 3, 4, 5];
const sum = numbers.reduce((acc, n) => acc + n, 0);
// 15
```

`reduce` は `map` や `filter` と比べて少し複雑です。第1引数のコールバックは「累積値（accumulator）」と「現在の要素」を受け取り、第2引数に初期値を指定します。

**`find` : 条件に合う最初の要素を返す（PHP 8.4 の `array_find` に相当）**

```javascript
// JavaScript
const users = [{ id: 1, name: '太郎' }, { id: 2, name: '花子' }];
const taro = users.find((user) => user.name === '太郎');
// { id: 1, name: '太郎' }
```

**`some` / `every` : 条件を満たす要素があるか / すべて満たすかを判定する**

```javascript
// JavaScript
const scores = [80, 65, 92, 45];
const hasPassed = scores.some((score) => score >= 90);  // true（92がある）
const allPassed = scores.every((score) => score >= 60);  // false（45がある）
```

### メソッドチェーン

JavaScript の配列メソッドが PHP の配列関数より読みやすい大きな理由が **メソッドチェーン** です。`map` や `filter` は新しい配列を返すので、続けて別のメソッドを呼び出せます。

```php
// PHP: ネストが深くなる
$result = array_map(
    fn($n) => $n * 2,
    array_filter($numbers, fn($n) => $n % 2 === 0)
);
```

```javascript
// JavaScript: 上から下に読める
const result = numbers
    .filter((n) => n % 2 === 0)
    .map((n) => n * 2);
```

PHP では内側の関数から外側に読む必要がありますが、JavaScript では上から下に処理の流れを追えます。

### LMS でよく使われるパターン

LMS のフロントエンドでは、配列の高階関数が至るところで使われています。実際のコードを見てみましょう。

**オブジェクト配列を UI 用のデータに変換する `map`:**

```tsx
// frontend/src/features/v2/userWorkspace/components/StatusChangeModal.tsx
const statusOptions = Object.values(USER_WORKSPACE_STATUS).map((status) => ({
  key: String(status.id),
  label: status.name,
}))
```

`Object.values()` でオブジェクトの値を配列に変換し、`map` でセレクトボックスの選択肢に必要な `{ key, label }` 形式に変換しています。このパターンは LMS 全体で頻繁に登場します。

**学習進捗を計算する `reduce`:**

```tsx
// frontend/src/app/v2/user/[workspaceId]/(main)/curriculums/[curriculumId]/chapters/[chapterId]/sections/[sectionId]/Client.tsx
const learned = sections.reduce((count, current) => (current.learned ? count + 1 : count), 0)
```

セクションの配列を走査し、`learned` が `true` の要素を数え上げています。初期値 `0` から始めて、条件に合う要素があるたびにカウントを1増やします。

**クエリパラメータを組み立てる `reduce`:**

```typescript
// frontend/src/lib/v2/fetch.ts
const params = Object.entries(queryParams || {}).reduce(
  (acc, [key, value]) => {
    if (Array.isArray(value)) {
      value.forEach((v, index) => {
        acc[`${key}[${index}]`] = String(v)
      })
    } else if (value !== undefined) {
      acc[key] = String(value)
    }
    return acc
  },
  // ...
)
```

`Object.entries()` でオブジェクトをキーと値のペアの配列に変換し、`reduce` でクエリ文字列用のオブジェクトに組み立てています。`reduce` のコールバックの第2引数で `[key, value]` と分割代入（後述）を使っている点にも注目してください。

## 分割代入

### 分割代入とは

**分割代入（Destructuring Assignment）** は、配列やオブジェクトから値を取り出して個別の変数に代入する構文です。PHP にも `list()` という似た機能がありますが、JavaScript の分割代入はより広範に使われます。

### 配列の分割代入

```php
// PHP
[$first, $second] = [1, 2];
// または
list($first, $second) = [1, 2];
```

```javascript
// JavaScript
const [first, second] = [1, 2];
```

PHP の `list()` と同じ発想です。配列の左辺に `[]` を使い、位置に対応する値を受け取ります。

LMS では、React の `useState` フックの戻り値を受け取るときに配列の分割代入が頻出します。

```tsx
// frontend/src/hooks/v2/useDisclosure.ts
const [isOpen, setIsOpen] = useState(defaultOpen)
```

```tsx
// frontend/src/hooks/v2/useAsyncSafeAction.ts
const [isLoading, setIsLoading] = useState(false)
```

`useState` は「現在の値」と「値を更新する関数」の2要素の配列を返します。配列の分割代入で、それぞれに意味のある名前を付けて受け取っています。このパターンは React 開発で最も頻繁に使う分割代入です。

### オブジェクトの分割代入

オブジェクトの場合は `{}` を使い、プロパティ名に対応する値を受け取ります。

```php
// PHP（PHP 8.x でも直接的な対応構文はない）
$user = ['name' => '太郎', 'age' => 25];
$name = $user['name'];
$age = $user['age'];
```

```javascript
// JavaScript
const user = { name: '太郎', age: 25 };
const { name, age } = user;
```

PHP では連想配列から値を取り出すのに1行ずつ書く必要がありますが、JavaScript ではオブジェクトの分割代入で一度に取り出せます。

LMS での実例を見てみましょう。

```typescript
// frontend/src/lib/v2/fetch.ts
const { pathParams, queryParams, requestBody } = params ?? {}
```

```typescript
// frontend/src/hooks/v2/auth.ts
const { data: user } = await fetchMeAsUser({}, {})
```

2つ目の例に注目してください。`{ data: user }` は「`data` プロパティの値を `user` という変数名で受け取る」という意味です。プロパティ名と異なる変数名を使いたいときに `:` で別名を指定できます。

### デフォルト値とネスト

分割代入にはデフォルト値を指定できます。

```javascript
// デフォルト値
const { name = '名無し', age = 0 } = {};
console.log(name); // '名無し'

// ネストした分割代入
const response = {
    data: {
        user: { name: '太郎' }
    }
};
const { data: { user: { name } } } = response;
console.log(name); // '太郎'
```

⚠️ **注意**: ネストが深い分割代入は読みにくくなります。実際の開発では、1〜2段階の分割代入に留めるのが一般的です。

### 関数の引数での分割代入

JavaScript では関数の引数でも分割代入を使えます。これは PHP にはないパターンで、LMS のコードでも頻繁に登場します。

```javascript
// 引数でオブジェクトを分割代入
function greet({ name, age }) {
    console.log(`${name}さんは${age}歳です`);
}

greet({ name: '太郎', age: 25 });
```

PHP では関数に多くのパラメータを渡すときに連想配列を使いますが、JavaScript ではオブジェクトの分割代入が自然な方法です。どのプロパティが渡されているかが呼び出し側のコードで一目で分かるという利点があります。

## スプレッド構文とレスト構文

### PHP の `...` 演算子との対比

PHP でも `...` 演算子は使われます。可変長引数や配列の展開に使う構文です。

```php
// PHP: 可変長引数
function sum(int ...$numbers): int {
    return array_sum($numbers);
}
sum(1, 2, 3); // 6

// PHP 7.4+: 配列の展開
$first = [1, 2];
$second = [3, 4];
$merged = [...$first, ...$second]; // [1, 2, 3, 4]
```

JavaScript の `...` も同じ記号を使いますが、より広い場面で活用されます。

### スプレッド構文（展開）

**配列のスプレッド:**

```javascript
const first = [1, 2];
const second = [3, 4];
const merged = [...first, ...second]; // [1, 2, 3, 4]
```

PHP の配列展開と同じです。

**オブジェクトのスプレッド:**

```javascript
const defaults = { color: 'red', size: 'M', weight: 100 };
const custom = { ...defaults, size: 'L' };
// { color: 'red', size: 'L', weight: 100 }
```

オブジェクトのスプレッドは PHP の連想配列では直接対応する構文がありません（PHP では `array_merge` を使います）。スプレッドで展開した後に同じキーのプロパティを書くと上書きされるため、「デフォルト値を設定しつつ、特定の値だけ変更する」というパターンで活用されます。

LMS ではストアの初期化やコンポーネントの Props 受け渡しでこのパターンが使われています。

```typescript
// frontend/src/store/v2/actor-store.ts
// Zustand ストアの初期化
{
  ...defaultInitState,
  setActor: (actor) => set({ actor }),
  setActorType: (actorType) => set({ actorType }),
  setManualLogout: () => set({ isManualLogout: true }),
  // ...他のアクション
}
```

```tsx
// frontend/src/components/v2/elements/Skeleton.tsx
export function Skeleton({ className, ...props }: Props) {
  return <HeroSkeleton className={className} {...props} />
}
```

この `Skeleton` コンポーネントのコードは2つのパターンを組み合わせています。引数部分の `{ className, ...props }` は **レスト構文**（後述）で、`className` を個別に取り出し、残りのプロパティをまとめて `props` に集めています。JSX 内の `{...props}` は **スプレッド構文** で、集めたプロパティを `HeroSkeleton` コンポーネントにそのまま展開して渡しています。

### レスト構文（残余）

スプレッド構文が「展開する」のに対し、レスト構文は「残りをまとめる」構文です。同じ `...` 記号を使いますが、用途が逆です。

**配列のレスト:**

```javascript
const [first, ...rest] = [1, 2, 3, 4, 5];
console.log(first); // 1
console.log(rest);  // [2, 3, 4, 5]
```

**オブジェクトのレスト:**

```javascript
const user = { name: '太郎', age: 25, email: 'taro@example.com' };
const { name, ...others } = user;
console.log(name);   // '太郎'
console.log(others); // { age: 25, email: 'taro@example.com' }
```

**関数の引数でのレスト:**

```javascript
function sum(...numbers) {
    return numbers.reduce((acc, n) => acc + n, 0);
}
sum(1, 2, 3); // 6
```

PHP の可変長引数 `function sum(int ...$numbers)` とほぼ同じです。

🔑 `...` が「左辺」（代入先、引数定義）にあればレスト構文、「右辺」（値、引数渡し）にあればスプレッド構文です。

| 位置 | 名称 | 動作 | 例 |
|---|---|---|---|
| 右辺（値として） | スプレッド | 展開する | `[...arr]`、`{...obj}` |
| 左辺（受け取り側） | レスト | まとめる | `const [a, ...rest] = arr` |

## ✨ まとめ

- **ES Modules** は PHP の `namespace` / `use` に相当する仕組みです。`export` で公開し、`import` で取得します。名前付きエクスポート（`{}`）とデフォルトエクスポート（`{}` なし）の2種類があります
- **npm / Yarn** は Composer に相当するパッケージマネージャーです。`package.json` が `composer.json`、`node_modules/` が `vendor/`、`yarn.lock` が `composer.lock` に対応します
- **配列の高階関数** は PHP の `array_map` / `array_filter` に相当しますが、メソッドチェーンで上から下に読める点が大きな違いです。`map` / `filter` / `reduce` / `find` / `some` / `every` を押さえておけば、LMS のコードの大半をカバーできます
- **分割代入** は配列やオブジェクトから値を取り出す構文です。React の `useState` の戻り値を受け取るパターンは特に重要です
- **スプレッド構文** は配列やオブジェクトを展開し、**レスト構文** は残りの要素をまとめます。どちらも `...` を使いますが、左辺にあればレスト、右辺にあればスプレッドです
